Object-Oriented Programming

The previous chapter introduced how R realises the content of mathematics and statistics textbooks. The present chapter shifts perspective and presents R’s foundational concepts from the standpoint of programming. On that basis, we take the first practical step in working with R: importing and exporting data — the two ends of any data analysis pipeline. We shall also examine how to interrogate basic features of a dataset, such as its size, its variable count, and its number of observations.

We begin by sketching R’s programming character.

Object-Oriented Programming (OOP) is a widely adopted architecture in modern computer programming. Compared with traditional procedure-oriented programming, OOP organises and encapsulates data so that more complex systems and demands can be handled cleanly. Many widely known languages — C++, Java, Ruby, Python — adopt OOP. R does the same, although in a rather idiosyncratic form (more on this when we discuss functions). Understanding OOP makes much of R’s behaviour easier to reason about.

OOP rests on two basic notions: classes and objects.

  • Class: a user-definable structure of an object, together with its properties and methods.
  • Object: a concrete instance of a particular class — much as “China” is to “country”, or “R” is to “letter”. Each instance carries the properties defined for its class.
  • Method: a command defined within a class that expresses some characteristic action of objects of that class. If a person eats, drinks, sleeps, and so on, then “eating, drinking, sleeping” are the person’s methods.
  • Property: an attribute used to describe some aspect of an object — hair colour, skin colour, height, weight, and so forth, for a person. All objects of a given class share the same set of properties, though their values may differ. Think of properties as the parameters of an object.

Beyond these, OOP languages typically share four additional traits.

  • Encapsulation. All properties and methods of an object are bundled together. Other objects cannot directly inspect or alter the inner state of an object — they may only influence it. One cannot turn an ectotherm into an endotherm, but one can affect a lizard’s body temperature by changing the ambient temperature of its tank.
  • Polymorphism. A single command can act differently on different object types. The most familiar example in R is summary(), which adapts its output to whatever class of object it receives.
summary(wvs7)
summary(wvs7$age)

Polymorphism in action

R’s polymorphism allows generic functions to handle object types they have not been explicitly programmed for. Beyond summary(), plot() is another classic polymorphic function. Apply plot() to two different kinds of object and observe how the output differs.

plot(wvs7$age)
plot(wvs7$age, wvs7$incomeLevel)

  • Inheritance. All subclasses of a parent class automatically possess the parent’s traits. By analogy: if humans have two legs, every individual human inherits that trait.
  • Safety. When a command is applied to an object, R checks the object’s class. If the class lies outside the command’s defined scope, R halts and returns an error.

Functions: Imperative Sentences with the Adverbial Postposed

R is built from two kinds of building blocks: functions and objects. A function can be thought of as an imperative sentence in which the adverbial phrase appears after the verb — exactly opposite to typical English word order. The general form is

<function_name>(<target_data>, <condition_1>, <condition_2>, ...)

The OOP creed expressed by this design is straightforward: do not touch the data; let the function act on it.

light <- function(finger){
  shadow <- finger + 5
  shadow
}
handShadow <- light(finger = 3)
handShadow

Packages

A package is a curated collection of functions — much like an app on a smartphone. A working snapshot of CRAN holds tens of thousands of registered packages (and many more on GitHub). Two common ways to install packages are:

install.packages("drhur")                          # from CRAN
devtools::install_github("sammo3182/drhur")         # from GitHub

The Assignment Arrow <-

The arrow <- is the assignment operator and is shorthand for the assign() function’s default behaviour. The general form is

<name> <- <object>

aValidObject <- 1:5
aValidObject
## [1] 1 2 3 4 5

->, =, <<-

R supports four ways to assign:

  1. assign()
  2. <-
  3. <<-
  4. =

Why <-

<- is preferred for several reasons:

  • Direction is intuitive. The arrow makes the data flow visually obvious — from the right-hand object into the left-hand name.
a <- 12
25 -> b   # the rightward arrow also works
  • No confusion with equality. Using <- for assignment leaves = and == unambiguous.
  • A keyboard shortcut exists.
    • PC: Alt + -
    • Mac: Option + -

When to Use Each

  • = — when you do not want to create a permanent object, for instance when supplying named arguments inside a function call.
median(y <- 1:10); y
median(x = 1:10); x
  • <<- — to refer to a variable in a parent scope, most often inside closures.
new_counter <- function() {
  i <- 0
  function() {
    # do something useful, then ...
    i <<- i + 1
    i
  }
}

Naming Rules

Object names in R must satisfy several constraints:

  1. They must not begin with a digit (incorrect: 1stday).
  2. Apart from . and _, no special characters are allowed (incorrect: M&M).
  3. Names are case-sensitive (X != x); recall that ! denotes “not” while != denotes “not equal”.
  4. Avoid overwriting built-in names (avoid list <- c(1:5)).
  5. Choose names that convey meaning.

Try creating one valid and one invalid object below.

# An invalid object name (begins with a digit)
# 5var_name <- data_frame(wvs7$education)


# A valid object name
var_name5 <- data_frame(wvs7$education)

Data Input

Built-in Datasets

R ships with a substantial collection of demonstration datasets, listed by data(). Many R packages also include their own datasets; once a package is loaded, its datasets become available in the same way.

data()

Pick a dataset listed by data(), load it, and use summary() to inspect its variables.

# An example
data(uspop)
summary(uspop)

Directly Readable Formats

Several common file formats can be read with base R alone:

  • .RDS — single R object
  • .RData — multiple R objects
  • .txt — plain-text tabular data
  • .csv — comma-separated values

Syntax: <name> <- <reader>(<path>)

df_rds <- readRDS("aDataset.rds")
df_txt <- read.table("D:/aDataset.txt")
df_csv <- read.csv("./aDataset.csv")

Reading via Packages

For other file formats, specialised packages offer dedicated readers. Load them with library or require and then call the appropriate function.

# SPSS, Stata, SAS
library(haven)
df_spss  <- read_spss("<FileName>.sav")
df_stata <- read_dta("<FileName>.dta")
df_sas   <- read_sas("<FileName>.sas7bdat")

# Tabular files (fast import)
library(readr)
df_csv   <- read_csv("<FileName>.csv")
df_table <- read_table("<FileName>.txt")

# Excel
library(readxl)
df_excel  <- read_excel("<FileName>.xls")
df_excel2 <- read_excel("<FileName>.xlsx")

# JSON
library(rjson)
df_json <- fromJSON(file = "<FileName>.json")

# XML / HTML
library(xml2)
df_xml  <- read_xml("<url>")
df_html <- rvest::html_table(rvest::read_html("<url>"))

The Swiss-army knife of data import is rio, whose import() function dispatches to the appropriate reader based on the file extension:

library(rio)
df_anything <- import("<AnyTypeOfData>")

Vectors

The function c() (“combine”) is the workhorse for constructing vectors. A vector’s elements must share the same data type — numeric, character, logical, factor, or otherwise.

Numeric Vectors

vec_integer <- c(1, -2, NA)
vec_double  <- c(1.5, -2.34, 1/3)

Two notes:

  1. NA stands for not available — R’s representation of a missing value.
  2. All elements of a single vector must share one type; if you mix types, R will silently coerce them to the most general one.

Generate a vector containing every even integer from 1 to 100.

# hint: help(seq)
seq(2, 100, by = 2)

Character Vectors

vec_chr <- c("R", "is", "powerful — and so are you.")

Generate the sequence of lowercase Roman letters from a to z.

vec_letters <- c("a", "b", "c", "d", "e")
letters[1:26]

Logical Vectors

vec_tf <- c(TRUE, TRUE, FALSE)
vec_tf
# c(TRUE, TRUE, FALSE) == c(1, 1, 0)

Suppose x is a vector containing (1, 1, 0). Convert it into a logical vector.

x <- c(1, 1, 0)
x <- as.logical(x)

Factor Vectors

A factor stores a categorical variable with a fixed set of allowed values (its levels). An ordered factor additionally records a meaningful order among the levels.

vec_fac <- factor(c(1, 2, 2, 3))
vec_ord  <- ordered(c(1, 2, 2, 3))
vec_fac2 <- factor(c(1, 2, 2, 3),
                   levels = c(3, 2, 1),
                   labels = c("Apple", "Pear", "Orange"))

When a new dataset arrives, the first task is to develop a rough sense of what is inside. Inspect the variables in wvs7:

str(wvs7)

Examine the class of wvs7$incomeLevel:

class(wvs7$incomeLevel)

List the values of incomeLevel and the frequency of each:

table(wvs7$incomeLevel)

Date and Time Vectors

R offers two representations of time:

  • as.POSIXct() — numeric input, integer storage; counts the seconds since the UNIX epoch (1970-01-01 00:00:00).
  • as.POSIXlt() — character input, list storage; each component (year, month, day, and so on) is an element of the list.
# The difference between as.POSIXct and as.POSIXlt
ct <- as.POSIXct("2023-03-20 10:11:12")
lt <- as.POSIXlt("2023-03-20 10:11:12")
unlist(ct)
## [1] "2023-03-20 10:11:12 CST"
unlist(lt)
##    sec    min   hour   mday    mon   year   wday   yday  isdst   zone gmtoff 
##   "12"   "11"   "10"   "20"    "2"  "123"    "1"   "78"    "0"  "CST"     NA
Sys.time()    # current date and time (with time zone)
today()       # today's date (year, month, day)
now()         # today's date and time
# CST is the time zone of the machine running this code

# The full pack
time1 <- Sys.time()
time2 <- as.POSIXlt(Sys.time())
time2$wday    # day of the week

## What if you only care about the date?
Sys.Date()
date1 <- as.Date("2019-01-02")
class(date1)

The Swiss-army knife for date handling is lubridate:

library(lubridate)

ymd("20221016")
mdy("10-16-2022")
dmy("16/10/2022")
ymd_hms("2022-10-16 09:00:00", tz = "Etc/GMT+8")
OlsonNames()

When date strings appear in inconsistent orders, such as

x <- c("20190101", "01012019", "021901")

how would you parse them?

# help(parse_date_time)
x <- c("20190101", "01012019", "021901")
parse_date_time(x, orders = c("ymd", "dmy", "dym"))

Arrays

An array is, as the name suggests, an “array of columns” that can store data of more than two dimensions. Use array() to create one.

# Create two vectors of different lengths
vector1 <- c(5, 9, 3)
vector2 <- c(10, 11, 12, 13, 14, 15)

# Feed them into an array
result <- array(c(vector1, vector2), dim = c(3, 3, 2))

Lists

Data Frames

A data frame is a special kind of list (or, equivalently, of matrix) in which:

  • columns are variables, each of equal length;
  • rows are observations.

In R:

df_toy <- data.frame(female = c(0, 1, 1, 0),
                     age    = c(29, 39, 38, 12),
                     name   = c("Iron Man", "Black Widow",
                                "Captain Marvel", "Captain America"))

df_toy

Data Attributes

  1. class, typeof — query the variable’s storage class and underlying type.
  2. nchar — get the length of a character string.
  3. levels — get or set the levels of a factor.
  4. nrow — return the number of rows of a matrix or data frame.
  5. ncol — return the number of columns of a matrix or data frame.
  6. dim — return the dimensions (the subspace spanned by the column vectors).
vec_integer <- c(1, -2, NA)

vec_double <- c(1.5, -2.34, 1/3)

vec_chr <- c("R", "is", "powerful — and so are you.")

Data Output

Syntax: <writer>(<object_to_save>, file = <path>)

Storing as Native R Data

saveRDS(df_toy, file = "df_toy.rds")
save(df_toy, ls_monks, file = "test.RData")

Storing as CSV

write.csv(df_toy, file = "toy.csv")

A note on encoding

If your dataset contains Chinese (or other non-Latin) text, write.csv() may produce mojibake (garbled characters) on some systems. Specifying fileEncoding = "UTF-8" or saving with readr::write_csv() typically resolves the issue.

You may export to Stata, SPSS, SAS, Excel, JSON, MATLAB, HTML, and many other formats — but should you? The native R formats are usually faster and more compact:
